# Видео-инструкции через Playwright

Превращает один Gherkin-сценарий в **тест + видео-инструкцию**. Тот же сценарий,
что проверяет фичу, записывается в браузере со slowMo и субтитрами по шагам.

## Когда применять
- Реализована фича → нужно отдать заказчику видео «как это работает».
- Нужно доказательство, что путь пользователя протестирован.
- Хочется живую документацию UI, которая не устаревает молча (сломается тест —
  не соберётся видео).

## Принципы (не нарушать)
1. **Видео — побочный продукт теста, а не отдельная процедура.** Сценарий пишется
   в любом случае; запись «прицепляется» флагом.
2. **Верификация — только по ассертам (`expect`).** Никогда не «смотреть» кадры
   записанного видео мультимодально — это дорого по токенам и не нужно. Видео для
   человека, ассерты для проверки. Исключение — не про верификацию: просмотр
   PNG из `screens/` как отдельная, явно запрошенная задача UX/UI-анализа
   (см. «Скриншоты прохода»).
3. **Стабильные селекторы (`data-testid`)**, а не хрупкие пути — иначе отладка
   съедает токены.
4. **Изолированные данные (seed)** через отдельный бэкенд/фикстуру — реальные
   данные не трогаем, `id`/состояние детерминированы.
5. **Тест-стенд на отдельных портах** — не конфликтовать с рабочим сервером
   разработчика.

## Разовая настройка проекта (один раз)
1. Создать каталог `qa/`, скопировать в него **только `.ts` из `lib/`** (caption,
   click-marker, record, fixtures, steps, screens — их импортируют спеки относительным
   путём, поэтому они обязаны жить в проекте), плюс `playwright.config.ts` и
   `package.json`. Python-озвучка (`narrate.py`, `accents.json`, `add-accent.py`)
   **не вендорится** — скрипты `video:voice*` зовут её из глобальной установки
   (`~/.claude/skills/video-walkthrough/`), словарь тоже глобальный; проектную
   лексику клади в `qa/lib/accents.json` (перекрывает глобальный, см. «Озвучка»).
2. `cd qa && npm install && npx playwright install chromium`. Для озвучки —
   разово `npm run setup` (голосовой стек в глобальный venv).
3. Настроить `webServer` в конфиге: поднять приложение (фронт+бэк) на **отдельных
   портах** с seed-данными; `reuseExistingServer: false` для детерминизма.
4. Добавить в приложение ~5 `data-testid` на ключевые узлы сценария (кнопки,
   карточки, поля ввода, строки списка). Селекторы по тексту/роли — там, где
   стабильно.
5. Завести seed-фикстуру (минимальные детерминированные данные) и `globalSetup`,
   сбрасывающий изменяемый слой перед прогоном.

## Авторинг сценария (каждая фича)
1. Создай пару файлов с именем `US-XX-<slug>` в `qa/scenarios/`:
   - `.feature` — человекочитаемый Gherkin (источник правды);
   - `.spec.ts` — исполняемый сценарий, шаги дословно зеркалят `.feature`.
   **Имя файла = имя ролика** на выходе (`qa/videos/US-XX-<slug>.mp4`).
2. Пиши шаги через обёртку `step(cap, 'текст шага', async () => { … })`:
   текст = строка Gherkin = заголовок в отчёте = субтитр на видео.
3. Действия — через хелперы (`openApp`, `dragSelect`, клики по `getByTestId`).
   Проверки — `expect(...)`.
4. Правила, выведенные практикой (соблюдай):
   - **Навигацию (`page.goto`) делай в первом шаге** — чтобы экран не был пустым;
     плашка субтитра переживает переход (восстанавливается на `load`).
   - **Не «смотреть» видео для проверки** — только ассерты.
   - **≥2 значимых элемента** там, где UI требует диапазон (напр. выделение слов).

## Прогон и выдача
1. Сначала быстро, без записи — проверить логику:
   `npm test`  (→ `playwright test --project=fast`)
2. Потом с записью:
   `npm run video`  (→ `--project=video`: slowMo + субтитры + mp4)
   Опция прожечь субтитры в кадр: `npm run video:burn` (`BURN_SUBS=1`).
3. Опционально озвучить субтитры голосом (см. «Озвучка»):
   `npm run video:voice` — Silero (офлайн); `npm run video:voice:hq` — edge
   Опционально собрать банк скриншотов прохода: `npm run screens`
   (см. «Скриншоты прохода»).
4. Отдай `qa/videos/US-XX-<slug>.mp4`. Он идёт в git как документация: одна фича =
   один стабильный файл, прогон перезаписывает его (историю ведёт git). Тяжёлые/
   многочисленные ролики → git-LFS (`*.mp4 filter=lfs`).

## Озвучка (опционально)
`npm run video:voice` (→ `python3 lib/narrate.py videos`) синтезирует речь по
`.srt` и подмешивает её в mp4 со сдвигами по таймкодам (ffmpeg `adelay`).
- Работает **пост-фактум по готовой паре mp4+srt** — старые видео можно озвучить
  без перегона теста; аудиодорожка заменяется целиком, повторный прогон идемпотентен.
- Движки (`--engine auto|silero|edge|say`). По умолчанию — Silero (`xenia`):
  офлайн, детерминированно. **«Качественная озвучка»** — `npm run video:voice:hq`
  (`--engine edge`, нейроголос Microsoft `Svetlana`): живая интонация по смыслу
  из коробки, латиницу и числа читает сам; требует интернет, API неофициальный.
  `say -v Milena` — аварийный, без установки. `auto` берёт Silero: torch ищется в
  текущем интерпретаторе, затем скрипт сам перезапускается в venv скилла
  (`~/.claude/skills/video-walkthrough/.venv`); если его нет — фолбэк на say.
  Venv создаётся один раз: `python3 -m venv ~/.claude/skills/video-walkthrough/.venv
  && …/.venv/bin/pip install torch omegaconf numpy ruaccent "transformers<5"
  num2words runorm` (модели скачаются в кэш при первом синтезе).
- **Ударения и термины.** Субтитры остаются чистыми; перед синтезом текст
  проходит препроцессор (`speech_text`): RUAccent расставляет ударения
  (`+` перед гласной, омографы по контексту), числа читаются словами, а
  словарь `lib/accents.json` задаёт произношение терминов кириллицей
  (`"Whisper": "у+испер"`). Silero **молча выбрасывает латиницу и цифры** —
  скрипт предупреждает о терминах без произношения: добавь их в accents.json
  и перепрогони. Словарные замены защищены от RUAccent (он стирает чужие `+`).
  Проектный `./lib/accents.json` (относительно места запуска, обычно `qa/lib/`)
  перекрывает глобальный — сюда проектную лексику, `narrate.py` при этом
  вызывается из глобальной установки.
  Незнакомую латиницу страхует runorm (транслит по токенам, пометка `ℹ` в
  выводе) — слово прозвучит, но для точного произношения всё равно добавь
  запись в словарь; `⚠` остаётся для случаев, где и runorm не справился.
- **Пополнение словаря — рабочий цикл.**
  1. Прогнал озвучку → в выводе `⚠ латиница без произношения` → добавь записи
     и перепрогони; заканчивай только когда предупреждений ноль.
  2. Пользователь услышал кривое слово (типичный случай — англицизмы кириллицей,
     где «е» читается мягко: тег→т+эг, флеш→фл+эш) → добавь запись и переозвучь
     **только затронутые ролики**: `grep -l слово qa/videos/*.srt`.
  3. Замена идёт по целым словам, поэтому русскому ключу нужны все формы слова.
     Не перечисляй их вручную — генерируй по базе словоформ (danakt/russian-words,
     1,5 млн слов, лежит в `lib/russian-words.txt` глобального скилла):
     `python3 lib/add-accent.py плеер пл+эер --apply` — найдёт и падежи, и
     композиты («видеоплеером», «медиаплееры»). **Сначала прогони без `--apply`
     и просмотри кандидатов**: короткие корни дают ложные совпадения («тег» →
     «стегать», «стратег») — для них `--no-compounds` и/или `--exclude`.
     Если базы нет (другая машина) — скрипт напечатает команду скачивания.
  4. Общеупотребимые термины (Enter, PDF, тег…) клади в глобальный словарь
     `~/.claude/skills/video-walkthrough/lib/accents.json`, проектную лексику
     (имена продуктов, файлов) — в локальный `qa/lib/accents.json`.
  Проверить произношение без записи видео: `speech_text('фраза')` из narrate.py.
- **Выразительность (silero).** Паузы по пунктуации (тире, двоеточие) SSML
  вставляет автоматически. Смысловые акценты — sidecar `<видео>.speech.json`
  рядом с srt: `{"текст субтитра": {"emphasis": ["ключевое слово"]}}` —
  ключевое слово читается чуть медленнее с микропаузами (эмуляция emphasis,
  которого у Silero нет). Заполняй при авторинге: один акцент на фразу —
  объект действия или результат шага; ключ дословно равен тексту субтитра.
- Фраза, не влезающая в свой 5-секундный слот, ускоряется (`atempo`, до x2);
  если не влезла и так — скрипт предупредит: поднимите `STEP_HOLD_MS` и
  перепишите видео.
- Токенов не тратит — чистое выполнение кода.

## Скриншоты прохода (опционально)
`npm run screens` (→ `SCREENS=1 playwright test --project=fast`) прогоняет те же
сценарии и складывает **чистые** кадры каждого шага в `qa/screens/<сценарий>/`:
пары `NN-before-…png` / `NN-after-…png` — состояние UI до и после действия,
имена из текстов шагов. Это банк скриншотов для UX/UI-анализа продукта.
- **Кадры чистые**: в fast-режиме плашка субтитра и маркер клика не рисуются —
  на скринах честный интерфейс, без оверлеев записи. Кадр «до» первого шага
  пропускается (до навигации страница пуста).
- **Перезапись как у видео**: прогон очищает каталог сценария и пишет набор
  заново; имена детерминированы (без таймстампов), историю ведёт git.
- **Скилл только собирает кадры, анализ — отдельная задача.** Сам прогон
  токенов не тратит (чистое выполнение кода). Смотреть PNG мультимодально —
  только когда пользователь явно попросил UX/UI-разбор; банк подходит как вход
  для скилла `feature-ux-analyze` («конкурент» = свой продукт).
- Изменения в спеках не нужны: съёмку делает та же обёртка `step()`.

## Настройки читаемости
- Субтитр держится на экране минимум **5 c** на шаг (`STEP_HOLD_MS`, по умолчанию
  5000). Быстрые шаги «дотягиваются» до минимума. Поменять:
  `STEP_HOLD_MS=7000 npm run video`.
- `slowMo` (в конфиге, ~300 мс) делает отдельные действия видимыми глазу.
- Клики подсвечиваются расходящимся кольцом в точке нажатия (`click-marker.ts`,
  только в режиме `video`) — зритель видит, куда нажал сценарий. Работает
  само, вшито в кадр, как субтитры.

## Стоимость по токенам
- Разовая настройка инфраструктуры: заметная, но один раз.
- Новая фича: +немного сверх обычного теста (сценарий и так пишется).
- Запись/конвертация/скриншоты — выполнение кода, ~0 токенов.
- ❌ Мультимодальный разбор кадров записанного видео — **никогда** (200k+ токенов
  на ролик, бессмысленно: проверяют ассерты). Просмотр PNG из `screens/` —
  только по явному запросу на UX/UI-анализ, не для верификации.

## Эталон
Смотри `scenarios/example.spec.ts` + `.feature` в этом репозитории как образец для

`steps.ts`).

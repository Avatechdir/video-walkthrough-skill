# Видео-инструкции через Playwright

Превращает один Gherkin-сценарий в **тест + видео-инструкцию**. Тот же сценарий,
что проверяет фичу, записывается в браузере со slowMo и субтитрами по шагам.

## Когда применять
- Реализована фича → нужно отдать заказчику видео «как это работает».
- Нужно доказательство, что путь пользователя протестирован.
- Хочется живую документацию UI, которая не устаревает молча (сломается тест —
  не соберётся видео).

## Принципы (не нарушать)
1. **Видео — побочный продукт теста, а не отдельная процедура.** Сценарий пишется
   в любом случае; запись «прицепляется» флагом.
2. **Верификация — только по ассертам (`expect`).** Никогда не «смотреть» кадры
   записанного видео мультимодально — это дорого по токенам и не нужно. Видео для
   человека, ассерты для проверки.
3. **Стабильные селекторы (`data-testid`)**, а не хрупкие пути — иначе отладка
   съедает токены.
4. **Изолированные данные (seed)** через отдельный бэкенд/фикстуру — реальные
   данные не трогаем, `id`/состояние детерминированы.
5. **Тест-стенд на отдельных портах** — не конфликтовать с рабочим сервером
   разработчика.

## Разовая настройка проекта (один раз)
1. Создать каталог `qa/`, скопировать в него `lib/` (caption, record, fixtures,
   steps), `playwright.config.ts`, `package.json` из этого репозитория.
2. `cd qa && npm install && npx playwright install chromium`.
3. Настроить `webServer` в конфиге: поднять приложение (фронт+бэк) на **отдельных
   портах** с seed-данными; `reuseExistingServer: false` для детерминизма.
4. Добавить в приложение ~5 `data-testid` на ключевые узлы сценария (кнопки,
   карточки, поля ввода, строки списка). Селекторы по тексту/роли — там, где
   стабильно.
5. Завести seed-фикстуру (минимальные детерминированные данные) и `globalSetup`,
   сбрасывающий изменяемый слой перед прогоном.

## Авторинг сценария (каждая фича)
1. Создай пару файлов с именем `US-XX-<slug>` в `qa/scenarios/`:
   - `.feature` — человекочитаемый Gherkin (источник правды);
   - `.spec.ts` — исполняемый сценарий, шаги дословно зеркалят `.feature`.
   **Имя файла = имя ролика** на выходе (`qa/videos/US-XX-<slug>.mp4`).
2. Пиши шаги через обёртку `step(cap, 'текст шага', async () => { … })`:
   текст = строка Gherkin = заголовок в отчёте = субтитр на видео.
3. Действия — через хелперы (`openApp`, `dragSelect`, клики по `getByTestId`).
   Проверки — `expect(...)`.
4. Правила, выведенные практикой (соблюдай):
   - **Навигацию (`page.goto`) делай в первом шаге** — чтобы экран не был пустым;
     плашка субтитра переживает переход (восстанавливается на `load`).
   - **Не «смотреть» видео для проверки** — только ассерты.
   - **≥2 значимых элемента** там, где UI требует диапазон (напр. выделение слов).

## Прогон и выдача
1. Сначала быстро, без записи — проверить логику:
   `npm test`  (→ `playwright test --project=fast`)
2. Потом с записью:
   `npm run video`  (→ `--project=video`: slowMo + субтитры + mp4)
   Опция прожечь субтитры в кадр: `npm run video:burn` (`BURN_SUBS=1`).
3. Отдай `qa/videos/US-XX-<slug>.mp4`. Он идёт в git как документация: одна фича =
   один стабильный файл, прогон перезаписывает его (историю ведёт git). Тяжёлые/
   многочисленные ролики → git-LFS (`*.mp4 filter=lfs`).

## Настройки читаемости
- Субтитр держится на экране минимум **5 c** на шаг (`STEP_HOLD_MS`, по умолчанию
  5000). Быстрые шаги «дотягиваются» до минимума. Поменять:
  `STEP_HOLD_MS=7000 npm run video`.
- `slowMo` (в конфиге, ~300 мс) делает отдельные действия видимыми глазу.

## Стоимость по токенам
- Разовая настройка инфраструктуры: заметная, но один раз.
- Новая фича: +немного сверх обычного теста (сценарий и так пишется).
- Запись/конвертация — выполнение кода, ~0 токенов.
- ❌ Мультимодальный разбор кадров записанного видео — **никогда** (200k+ токенов
  на ролик, бессмысленно: проверяют ассерты).

## Эталон
Смотри `scenarios/example.spec.ts` + `.feature` в этом репозитории как образец для
копирования. Библиотека — в `lib/` (`caption.ts`, `record.ts`, `fixtures.ts`,
`steps.ts`).
